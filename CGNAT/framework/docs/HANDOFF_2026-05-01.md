### What Is Not Fully Fixed

#### Router 1 inbound-initiated ping is still not proven

The unresolved requirement is:

- `194.138.36.86 -> 10.20.30.10`
- Router 1 must answer with a fresh echo-reply

This is not the same as:

- Router 1 sending a ping to `194.138.36.86`
- and receiving the reply to its own outbound flow

The strongest read at the end of the session:

- the tunnel and return path are fine
- the remaining problem is Router 1 local host modeling for unsolicited inbound traffic to `10.20.30.10`

Strong clues:

- `XfrmInTmplMismatch` was repeatedly the key kernel symptom
- `ip route get 10.20.30.10 from 194.138.36.86 iif cgxfrm-r1`
  returned:
  - `RTNETLINK answers: Invalid cross-device link`

That suggests Router 1 still does not have a clean local-delivery model for `10.20.30.10` when packets arrive from `cgxfrm-r1`.

#### Router 2 corrected model is not complete

Router 2 still needs cleanup on the live backend responder side.

Current live failure:

  - `10.250.1.11[500] -> 23.20.31.151[500]`

- backend actually sees the packet as:

- which means the final live routing/fronting path for Router 2 is still not aligned the way Router 1 was aligned

### Most Important Conclusions

1. `route_localnet` is not the right knob for this issue.
   - It is about routing `127.0.0.0/8` on non-loopback interfaces.
   - Our issue is with:
     - `10.20.30.10`
     - `10.250.1.10`

2. `rp_filter` is not the main blocker here.

3. The real fixes were:
   - corrected inner endpoint model
   - routed VIP / muxer path for loopback-sourced inner peers
   - `disable_policy=1` on the inner xfrm interface for Router 1

4. The remaining issue is not the tunnel path.
   - It is the local host / delivery model on Router 1 for unsolicited inbound traffic.
